**#** **编程范式与编码规范**



本文件定义了所有项目必须遵守的编程范式、编码规范和开发流程。



\---



**##** **一、核心编程范式（四位一体）**



**###** **1. SDD — 规范驱动开发（最外层，贯穿全程）**



**核心原则：规范是第一产物，代码是派生物。**



开发流程：

\1. **指定（Specify）**：先写清楚规范文档（需求、设计、接口、数据结构）

\2. **生成（Generate）**：根据规范生成/编写代码

\3. **验证（Validate）**：验证代码是否符合规范，不符合则修正规范或代码



规范文档要求：

\- 文档必须在写代码前确认完毕

\- 写完代码后必须回头检查文档是否需要同步更新

\- 规范要精确、可机读、无歧义



**###** **2. DDD — 领域驱动设计（架构层面）**



**核心原则：围绕业务领域建模，而不是围绕技术。**



后端分层架构（六边形架构）：

\```

domain/      # 领域层（核心业务逻辑，不依赖任何框架）

 entity/     # 实体（有唯一标识的业务对象）

 valueobject/  # 值对象（无标识，不可变）

 aggregate/   # 聚合根（一致性边界）

 repository/   # 仓储接口（只定义接口，不实现）

 service/    # 领域服务（跨实体的业务逻辑）

 event/     # 领域事件



application/   # 应用层（用例编排，协调领域对象）

 service/    # 应用服务（对外暴露用例）

 dto/      # 数据传输对象

 command/    # 命令对象（写操作）

 query/     # 查询对象（读操作）



infrastructure/  # 基础设施层（技术实现细节）

 persistence/  # 数据库实现（实现 domain/repository 接口）

 config/     # 配置类

 external/    # 外部服务（邮件、第三方 API 等）



interfaces/    # 接口层（对外暴露）

 rest/      # REST 控制器

 dto/      # 请求/响应 DTO

\```



关键规则：

\- Domain 层不依赖 Spring、MyBatis 等任何框架

\- 依赖方向：interfaces → application → domain ← infrastructure

\- 每个聚合根负责维护自身的一致性



**###** **3. TDD — 测试驱动开发（编码层面）**



**核心原则：先写测试，再写实现，最后重构。**



红绿重构循环：

\1. **红（Red）**：写一个失败的测试

\2. **绿（Green）**：写最少的代码让测试通过

\3. **重构（Refactor）**：在测试保护下优化代码



测试分层：

\- **单元测试**：测试单个类/方法，Mock 所有依赖（JUnit 5 + Mockito / Vitest）

\- **集成测试**：测试多个组件协作（Spring Boot Test / @DataJpaTest）

\- **端到端测试**：测试完整用户流程（可选）



测试命名规范：

\```java

*// 格式：方法名_场景_期望结果*

@Test

void findById_whenWordExists_returnsWord() { }



@Test

void save_whenUserNotFound_throwsException() { }

\```



覆盖率要求：

\- 核心业务逻辑（domain 层）：≥ 80%

\- 应用服务层：≥ 70%

\- 工具类：≥ 60%



**###** **4. BDD — 行为驱动开发（验收层面）**



**核心原则：用业务语言描述功能，确保代码行为符合业务预期。**



用户故事格式：

\```

作为 [用户角色]

我希望 [完成某个功能]

以便 [获得某个价值]

\```



验收条件格式（Given/When/Then）：

\```

Given（前置条件）：用户已登录，选择了基础单词书

When（触发动作）：用户点击"开始学习"

Then（期望结果）：系统展示第一张单词卡，显示印尼语单词

\```

**##** **二、编码规范**



**###** **代码注释**

\- 所有代码必须加**中文注释**

\- 类注释：说明类的职责和所属领域

\- 方法注释：说明参数、返回值、业务含义

\- 复杂逻辑：行内注释解释为什么这样做



**###** **代码规模限制**

\- 单个方法：不超过 **50 行**

\- 单个文件：不超过 **1000 行**

\- 单个 Vue 组件：不超过 **800 行**

\- 超出时必须拆分为更小的方法/文件/组件



**###** **模块化原则**

\- 单一职责：每个类/函数只做一件事

\- 最大嵌套深度：3 层

\- 禁止注释掉的死代码提交



**###** **依赖管理**

\- 不得随意引入新依赖，需先确认

\- 优先使用项目已有依赖

\- 优先使用语言内置特性



\---



**##** **三、UI/UX 规范**



\- 使用 ui-ux-pro-max 工具生成设计系统（`python3 steering/ui-ux-pro-max/scripts/search.py`）

\- 界面友好、简洁、美观

\- 支持日间/夜间模式

\- 响应式设计，兼容 PC 和移动端

\- 所有可点击元素必须有 `cursor-pointer`

\- 禁止使用 emoji 作为 UI 图标，使用 SVG 图标库



\---



**##** **四、测试规范**



\- 写完代码必须有对应测试

\- 测试必须在提交前全部通过

\- 禁止为了让测试通过而 Mock 核心业务逻辑

\- 发现 Bug 先写复现测试，再修复



\---



**##** **五、知识沉淀规范**



\- 遇到的难题和解决方案必须记录为 `.kiro/steering/` 或 `.kiro/skills/` 文件

\- 文件命名：`problem-{问题简述}.md`

\- 内容包含：问题描述、根因分析、解决方案、预防措施

\- 适用范围：所有项目



\---



**##** **六、文档规范**



\- 规范文档先于代码存在

\- 代码完成后同步检查文档是否需要更新

\- API 接口必须有完整的请求/响应示例

\- 数据库表必须有字段说明注释
